## 1. auto_verify.py (оркестратор)

### Сильные стороны
- Полный docstring с примерами использования
- Грамотная архитектура: subprocess-изоляция каждого скрипта
- Таймаут 60 сек для subprocess
- Стандартизированный формат выхода (JSON)
- Пресеты полностью покрывают все скрипты
- Проверка существования файлов на входе

### Находки

**🟢 РЕКОМЕНДАЦИЯ** `auto_verify.py:100` — Type hints есть для `list[str]`, но не для возвращаемого значения `run_script`.
**Рекомендация:** добавить `-> dict` к `run_script`.

**🟢 РЕКОМЕНДАЦИЯ** `auto_verify.py:166` — При `result.returncode == 0` и пустом stdout результат трактуется как error. Если скрипт правильно отработал, но вывел пустую строку — ложноотрицательный.
**Рекомендация:** отдельная обработка пустого stdout с returncode 0.

**🟢 РЕКОМЕНДАЦИЯ** `auto_verify.py:78-97` — Пресеты М1, М5, П6, П9 содержат пустые списки скриптов. Если пользователь указывает `--preset М1`, ни один скрипт не запустится (тихо).
**Рекомендация:** вывести предупреждение при пустом пресете.

## 2. verify_docx_format.py (формат docx)

- Допуски (TOLERANCE_CM, TOLERANCE_PT) — грамотный подход
- Конвертация EMU через отдельные функции с обработкой None

### Находки

**🟡 СУЩЕСТВЕННО** `verify_docx_format.py:147` — Проверяется только первый `run` в абзаце (`break` на строке 147). Если абзац содержит несколько run с разными шрифтами (например, часть жирная, часть курсив), остальные runs будут проигнорированы.
**Рекомендация:** проверять все runs или хотя бы первый непустой.

**🟡 СУЩЕСТВЕННО** `verify_docx_format.py:85-148` — Не проверяется межстрочный интервал (`line_spacing: 1.15` из стандарта определён, но нигде не используется).
**Рекомендация:** добавить проверку `para.paragraph_format.line_spacing`.

**🟡 СУЩЕСТВЕННО** `verify_docx_format.py:85-148` — Не проверяется отступ первой строки (`first_line_indent_cm: 1.5` определён в STANDARD, но не проверяется).
**Рекомендация:** добавить проверку `para.paragraph_format.first_line_indent`.

**🟡 СУЩЕСТВЕННО** `verify_docx_format.py:85-148` — Не проверяется `space_after_pt: 8` из стандарта.
**Рекомендация:** добавить проверку `para.paragraph_format.space_after`.

**🟢 РЕКОМЕНДАЦИЯ** `verify_docx_format.py:63-82` — Колонтитулы (`header_font_size_pt`, `footer_font_size_pt`) определены в STANDARD, но не проверяются.
**Рекомендация:** добавить функцию `check_headers_footers`.

**🟢 РЕКОМЕНДАЦИЯ** `verify_docx_format.py:155-173` — Таблицы: проверяется только первая строка первой ячейки первой таблицы (`break` на строках 170-173 создают четырёхуровневый ранний выход).
**Рекомендация:** проверять хотя бы заголовочную строку полностью.

## 3. verify_sums.py (суммы xlsx)

### Сильные стороны
- Три типа проверок: суммы, границы, отрицательные значения
- Ключевые слова для итогов на двух языках (рус + англ)
- Порог diff > 0.01 для float-сравнения — корректно
- Стандартизированный вывод

### Находки

**🟡 СУЩЕСТВЕННО** `verify_sums.py:133` — `openpyxl.load_workbook(filepath, data_only=True)` загружает весь файл в RAM. Для очень больших xlsx (100k+ строк) возможен OOM.
**Рекомендация:** для больших файлов использовать `read_only=True` или `openpyxl.load_workbook(filepath, data_only=True, read_only=True)`.

**🟡 СУЩЕСТВЕННО** `verify_sums.py:29-37` — Merged cells не обрабатываются. Если итоговая ячейка объединена, `cell.value` может быть None для вторичных ячеек объединения.
**Рекомендация:** добавить проверку `ws.merged_cells.ranges`.

**🟡 СУЩЕСТВЕННО** `verify_sums.py:29-37` — Скрытые строки (`ws.row_dimensions[row].hidden`) не учитываются при суммировании. Если скрытые строки содержат данные, сумма может не совпадать по дизайну.
**Рекомендация:** добавить параметр/флаг для пропуска скрытых строк.

**🟢 РЕКОМЕНДАЦИЯ** `verify_sums.py:160` — Двойной вызов `find_sum_rows(ws)`: один в цикле (строка 138), второй в строке 160 для details. Лишняя работа.
**Рекомендация:** кэшировать результат.

**ℹ️ ИНФОРМАЦИЯ** `verify_sums.py:22` — Символ `σ` в SUM_KEYWORDS — нестандартное обозначение. Вероятность встретить мала, но не навредит.

## 4. verify_references.py (битые ссылки)

### Сильные стороны
- 8 паттернов ссылок: таблица, рисунок, диаграмма, приложение, пункт, слайд, схема, график
- Двусторонняя проверка: ищет ссылки И цели, потом сверяет
- Контекст (30 символов вокруг) для удобства анализа

### Находки

**🟡 СУЩЕСТВЕННО** `verify_references.py:17` — Паттерн `Рисунок`: `r"(?:см\.?\s*)?(?:рисун[окка]+\s*)(\d+)"` — наивная морфология. Не совпадёт с "рисунке", "рисунков" и другими формами.
**Рекомендация:** использовать `рисун\w*` для более полного совпадения.

**🟢 РЕКОМЕНДАЦИЯ** `verify_references.py:27-37` — Весь текст конкатенируется в одну строку `full_text = "\n".join(...)`. Для очень больших документов (1000+ страниц) это загрузит всю строку в RAM и прогонит все паттерны.
**Рекомендация:** для больших документов обрабатывать абзацы поблочно.

**🟢 РЕКОМЕНДАЦИЯ** `verify_references.py:46-47` — Таблицы считаются по порядку (`i + 1`), но на практике таблицы в docx часто нумеруются через подписи (captions), а не по порядковому номеру. Возможен ложноположительный при удалении таблицы из середины.
**Рекомендация:** приоритет отдать подписям, fallback на порядковый номер.

**ℹ️ ИНФОРМАЦИЯ** `verify_references.py:7` — Множественный import на одной строке (`import argparse, json, re, sys`). Стилистическое замечание (PEP 8 рекомендует раздельные imports).

## 5. verify_cross_file.py (кросс-файловая согласованность)

### Сильные стороны
- Три типа сверки: точное значение, семантическая группировка, ФИО
- Ленивый импорт зависимостей — скрипт не падает, если нет python-docx, но нет xlsx файлов
- Ограничение выдачи ФИО ([:5]) — защита от шума

### Находки

**🟡 СУЩЕСТВЕННО** `verify_cross_file.py:26-27` — Паттерн числа `r'(\d[\d\s,.]+\d)'` имеет проблему: минимум 2 цифры. Одноцифровые числа (например, "5 человек") не будут найдены.
**Рекомендация:** добавить паттерн для одноцифровых чисел с контекстом.

**🟢 РЕКОМЕНДАЦИЯ** `verify_cross_file.py:37-38` — ФИО: паттерн `([А-ЯЁ][а-яё]+)\s+([А-ЯЁ][а-яё]+)\s+([А-ЯЁ][а-яё]+)` совпадает с любыми тремя словами с заглавной буквы. Ложноположительные на фразах типа "Российская Федерация Правительство".
**Рекомендация:** добавить фильтр по стоп-словам или валидацию формата ФИО.

**🟢 РЕКОМЕНДАЦИЯ** `verify_cross_file.py:90-99` — Функция `verify` не имеет type hints.

## 6. verify_dates.py (валидация дат)

### Сильные стороны
- Три формата дат: "15 марта 2025", "01.04.2025", "2025-06-15"
- Прогнозный контекст: даты в будущем не маркируются ошибочными, если контекст содержит "план", "прогноз" и т.д.
- Проверка реалистичности (1990-2030)

### Находки

**🟡 СУЩЕСТВЕННО** `verify_dates.py:76` — Жёстко захардкожены границы `1990–2030`. Сейчас 2026, и документ с датами 2027-2030 — это ближайшее будущее, которое может быть совершенно нормально. Верхняя граница слишком близка.
**Рекомендация:** сделать границы параметризуемыми или расширить до `datetime.now().year + 10`.

**🟢 РЕКОМЕНДАЦИЯ** `verify_dates.py:60-61` — `future_years` содержит строки `str(now.year + i)`, которые добавляются к `prognosis_kw`. Но при этом год "2026" (текущий) тоже включён, что означает: любая дата 2026 в будущем никогда не будет помечена. Это может быть как плюсом, так и минусом.

**🟢 РЕКОМЕНДАЦИЯ** `verify_dates.py:19-34` — Функция `extract_dates` не имеет type hints.

**ℹ️ ИНФОРМАЦИЯ** `verify_dates.py:52` — При чтении xlsx все значения конвертируются через `str(c)`. Даты, которые openpyxl уже распарсил как `datetime`, будут преобразованы в строку формата "2025-03-15 00:00:00", которая не совпадёт ни с одним паттерном. Потенциальный пропуск дат из xlsx.
**Рекомендация:** обрабатывать `datetime` объекты из openpyxl напрямую.

## Межскриптовое взаимодействие

### Формат вывода

Все 11 скриптов используют **единый формат JSON-ответа**:
```json
{
  "script": "имя_скрипта.py",
  "status": "pass|warn|fail|skip|info|error",
  "details": "строка",
  "items_checked": int,
  "items_passed": int,
  "items_warned": int,
  "items_failed": int,
  "findings": [{"severity": "...", "location": "...", "description": "..."}]
}
```
**Вердикт:** формат полностью совместим с `generate_report.py` и `auto_verify.py`.

### Циклические зависимости

Нет. Каждый скрипт работает автономно. `auto_verify.py` вызывает остальные через subprocess, а не через import.

### Python-совместимость

Все скрипты используют Python 3.11+. Используются:
- `list[str]` (auto_verify.py:100) — требует Python 3.9+
- f-strings — Python 3.6+
- `Path` — Python 3.4+

**Вердикт:** единый стандарт Python 3.9+.
