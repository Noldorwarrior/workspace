# Промт: конвертация документов верификации в скилл для Claude.ai проекта

## Роль

Ты — инженер скиллов для Claude.ai. Твоя задача — преобразовать набор документов системы верификации в структурированный набор файлов (скилл), готовый к загрузке в проект Claude.ai.

## Целевая среда

**Claude.ai** — чат с проектом (Project Knowledge). Это значит:

- Файлы загружаются пользователем в Project Knowledge вручную
- Нет субагентов, нет bash — только режим `self` (Claude проверяет сам)
- Скрипты Python могут запускаться через computer use (если доступен), но не гарантированно
- Взаимодействие с пользователем: ask_user_input (интерактивный выбор) или обычные вопросы в чате
- SKILL.md загружается как Project Instructions (системный промт проекта)
- Reference-файлы загружаются в Project Knowledge и доступны через project_knowledge_search

**Следствия для архитектуры скилла:**

1. `agent_patterns.md` — упростить: убрать паттерны субагентов/оркестратора, оставить только режим `self` с описанием workflow самопроверки. Сохранить раздел антипаттернов. Добавить примечание: «При использовании в Claude Code активируются режимы subagent/parallel/orchestrator — см. полную документацию.»
2. Скрипты — давать инструкции для ручного запуска через computer use (bash_tool), но workflow не должен ломаться без них. Каждый скриптовый механизм должен иметь fallback-инструкцию для LLM-проверки вручную.
3. Контрольная точка — использовать ask_user_input с вариантами.
4. SKILL.md — писать как Project Instructions: Claude.ai загрузит его целиком в контекст.

## Контекст

У тебя есть набор файлов — система верификации результатов работы LLM (32 механизма проверки, 19 пресетов, Python-скрипты автоматизации + LLM-проверки). Система предназначена для проверки документов, расчётов, презентаций, аналитических записок и других результатов работы.

### 2. Тело SKILL.md (≤500 строк)

Включить:

1. **Архитектура** — дерево файлов скилла с пояснениями. Указать: «SKILL.md = Project Instructions, references/ = Project Knowledge, scripts/ = опционально для computer use».

2. **Загрузка reference-файлов** — таблица: при каком сценарии какой файл искать через `project_knowledge_search`. Правила:
   - Конкретные механизмы по номерам → `project_knowledge_search` по номеру механизма
   - Пресет М1–М5 или П1–П14 → искать `presets_matrix` → затем нужные `mechanisms_*`
   - П5 «Максимум» → все `mechanisms_*` файлы
   - Нужна логика самопроверки → искать `verification_self_mode`
   - Нужен скрипт → если доступен computer use, запустить из `scripts/`; иначе — выполнить LLM-проверку вручную по процедуре из reference-файла

3. **Карта механизмов → файлы** — таблица: файл → список номеров механизмов.

4. **Разделение: скрипты vs LLM** — две таблицы:
   - Автоматизируемые скриптами (если доступен computer use): № механизма → скрипт
   - LLM-проверка (всегда доступна): № механизма → краткая инструкция

5. **Основной workflow** (5 этапов):
   - Этап 0: автодетекция типа задачи (тип, входные/выходные файлы, адресат)
   - Этап 1: автоподбор пресета (П1–П14) + малой верификации (М1–М5)
   - Этап 2: 🔒 КОНТРОЛЬНАЯ ТОЧКА — показать план через ask_user_input, дождаться одобрения
   - Этап 3: выполнение основной задачи (верификация ПОСЛЕ, не во время)
   - Этап 4: запуск верификации — скрипты (если computer use доступен) + LLM-проверки
   - Этап 5: блок «Результаты верификации» с таблицами и уровнем уверенности

6. **Быстрые сценарии** — 5 типовых:
   - Аналитическая записка → П1 + М2
   - Презентация pptx + html → М4 + элементы П7
   - Бюджет/KPI → П3 + М2
   - Мероприятие/сценарий → П8 + М3
   - Аудит чужого документа → П13

### 3. Контрольная точка (Этап 2)

**Формат сообщения:**
```
📋 План верификации

Рекомендуемый пресет: [название]
Обоснование: [почему именно этот]

  ☑ [список скриптов]
```

**Варианты для ask_user_input:**
- «Всё ок — начинай»
- «Добавить механизмы — выберу»
- «Убрать лишнее — выберу»
- «Сменить пресет»

**Правила:**
- НИКОГДА не пропускать контрольную точку

- Если ответ неясен → переспросить
- Одного подтверждения достаточно, не переспрашивать «вы уверены?»

## Требования к reference-файлам

### mechanisms_*.md (6 файлов)

Каждый файл содержит:

1. **Оглавление** — список механизмов с якорными ссылками
2. **Для каждого механизма:**
   - **Суть** (1–2 предложения)
   - **Когда применять** (список ситуаций)
   - **Пошаговая процедура** (шаги с инструкциями)
   - **Пример** (конкретный, с таблицами, из реальных кейсов)
   - **Шаблон фиксации** (формат отчёта по механизму)
   - **Маркировка:** скрипт / скрипт+LLM / LLM-only
   - **Fallback для Claude.ai:** если механизм обычно автоматизируется скриптом, но computer use недоступен — что делать LLM вручную (краткая инструкция, 2–3 предложения)
3. **Примеры** из реальных контекстов:
   - Госполитика РФ (молодёжная политика, нацпроекты, Росмолодёжь, РДДМ)
   - НИЯУ МИФИ (ЦПИКП, трансформация)
   - Т-Банк (DIGITAL, инвестиции)
   - Конгресс/мероприятия (спикеры, тайминги, площадки)

**Перенос содержания:** все механизмы, процедуры, примеры и шаблоны фиксации из исходных файлов переносятся ПОЛНОСТЬЮ, без сокращений. Добавляется только поле «Fallback для Claude.ai».

### presets_matrix.md

Перенести полностью без изменений:
1. Таблицы М1–М5 и П1–П14 с механизмами и скриптами
2. Полную кросс-матрицу (механизмы × пресеты) — 32 строки × 19 столбцов
3. Правила комбинирования пресетов
4. Таблицу автоподбора пресета по типу задачи (ключевые слова → пресет → малая)

### verification_self_mode.md (замена agent_patterns.md)

Новый файл, адаптированный под Claude.ai:

1. **Режим self** — описание: Claude проверяет свою работу сам, без субагентов. Ключевой риск: confirmation bias (склонность подтверждать собственную правоту).

2. **Приёмы против confirmation bias:**
   - Перечитай документ «глазами критика» — представь, что ты НЕ автор
   - Проверяй по процедуре, не по памяти — открой reference-файл и следуй шагам
   - Не пропускай шаги процедуры, даже если «очевидно, что всё ок»
   - Если нашёл 0 ошибок — это подозрительно, перепроверь самый рискованный механизм

3. **Fallback-инструкции для скриптовых механизмов:**
   Для каждого скриптового механизма — что делать, если computer use недоступен:
   | Скрипт | LLM-fallback |
   | verify_docx_format | Проверить шрифт, поля, заголовки визуально по чеклисту |
   | verify_sums | Пересчитать итоги вручную |
   | verify_references | Найти все «см.», «Таблица N», «Приложение X» и проверить цели |
   | verify_cross_file | Составить реестр общих точек данных, сверить попарно |
   | verify_numbering | Пройти по всем нумерованным элементам последовательно |
   | verify_dates | Выписать все даты, построить шкалу, проверить порядок |
   | verify_pptx_format | Проверить 16:9, размер шрифтов, количество текста |
   | verify_pptx_html_sync | Сравнить слайды попарно: N pptx ↔ секция N html |
   | diff_versions | Сравнить два текста, выписать все отличия |
   | generate_report | Сформировать отчёт в формате markdown по шаблону |

4. **Workflow самопроверки в Claude.ai:**
   ```
   Создал документ
   ↓
   СТОП. Не отправляй пользователю.
   ↓
   Открой presets_matrix.md → найди свой пресет
   ↓
   Открой нужный mechanisms_*.md → следуй процедуре каждого механизма
   ↓
   Если computer use доступен → запусти скрипты
   ↓
   Сформируй блок «Результаты верификации»
   ↓
   Отправь пользователю: документ + результаты верификации
   ```

5. **Антипаттерны** (перенести из agent_patterns.md):
   - ❌ Верификация во время создания
   - ❌ Формальная отписка «Всё ок ✅» без деталей
   - ❌ Пропуск контрольной точки
   - ❌ Автоматическое исправление без уведомления
   - ❌ Повторная верификация без изменений (использовать П14 «Итератор»)

6. **Примечание для Claude Code:**
   > При использовании в Claude Code доступны расширенные режимы: subagent (независимый верификатор), parallel (2–3 параллельных субагента), orchestrator (полная оркестрация П5 «Максимум»). Документация: см. исходный agent_patterns.md.

## Требования к скриптам

### Адаптация путей

1. Все скрипты переносятся в `scripts/` **без изменения логики**.
2. **Единственные изменения:**
   - В `auto_verify.py`: исправить импорты — заменить `import verify_sums` на `from . import verify_sums` или добавить `sys.path.insert(0, os.path.dirname(__file__))` в начало файла.
   - В каждом скрипте: убедиться, что `if __name__ == "__main__":` работает при запуске из любой директории.
   - Пути к другим скриптам внутри `auto_verify.py`: использовать `Path(__file__).parent` вместо абсолютных путей.

3. **requirements.txt:**
   ```
   python-docx>=0.8.11
   openpyxl>=3.1.0
   ```

4. Каждый скрипт должен иметь docstring на русском с:
   - Описание: что проверяет
   - Использование: `python verify_sums.py --file data.xlsx`
   - Выход: JSON

### Проверка работоспособности

После адаптации путей проверь, что:
- `python scripts/auto_verify.py --help` работает
- `python scripts/verify_docx_format.py --help` работает
- Импорты между скриптами внутри `scripts/` не ломаются

## Правила обработки

1. **Не теряй содержание.** Каждый механизм, пресет, пример, процедура, шаблон фиксации из исходных файлов должен быть перенесён в результат. При конвертации `agent_patterns.md` → `verification_self_mode.md` сохрани антипаттерны и добавь примечание о Claude Code.

2. **Не выдумывай.** Не добавляй новых механизмов, пресетов, примеров. Используй только то, что есть в исходных файлах. Единственные добавления: fallback-инструкции для Claude.ai и поле «Fallback для Claude.ai» в mechanisms_*.md.

3. **Структура.** YAML-фронтматтер обязателен в SKILL.md. Reference-файлы — чистый markdown без фронтматтера. Скрипты — Python с docstring.

4. **Язык — русский.** Все инструкции, примеры, комментарии в коде — на русском. Допустимы технические термины: workflow, YAML, JSON, fallback, computer use, ask_user_input.

5. **SKILL.md ≤ 500 строк.** Это системный промт — он загружается целиком. Перегруз = потеря качества. Детали — в reference-файлах, доступных через project_knowledge_search.

6. **Скрипты — минимальные правки.** Содержание скриптов НЕ меняется. Адаптируются только: (а) пути импортов внутри `scripts/`, (б) добавляется `requirements.txt`.

7. **Progressive disclosure.** Claude.ai загружает SKILL.md в контекст, но reference-файлы — нет. Они доступны через `project_knowledge_search`. SKILL.md должен чётко указывать: «ищи [что] когда [когда]».

8. **Fallback-first.** Каждый скриптовый механизм должен работать без скрипта. Скрипт — ускорение и формализация, но не обязательное условие.

## Порядок выполнения

1. Создай директорию `verification/`
2. Напиши `SKILL.md` (маршрутизация, workflow, карта) — ≤500 строк
3. Создай `references/` и перенеси:
   - 6 файлов `mechanisms_*.md` — полностью + добавь fallback-поля
   - `presets_matrix.md` — полностью без изменений
   - Создай `verification_self_mode.md` на основе `agent_patterns.md`
4. Создай `scripts/`:
   - Перенеси все 11 .py файлов, исправив импорты
   - Создай `requirements.txt`
5. Создай `INSTALL.md` (инструкция по установке + примеры использования)
6. Проведи 5 тестов корректности (см. раздел «Тесты»)
7. Выведи финальное дерево файлов и чеклист тестов

## Выходные файлы

Все файлы должны быть созданы и записаны. В конце выведи:
- Дерево файлов с размерами
- Количество строк в SKILL.md (≤500)
- Чеклист тестов (5 пунктов)
- Таблицу полноты: 32 механизма × (reference + пресеты + скрипт + fallback)
