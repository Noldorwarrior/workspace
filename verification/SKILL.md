# Скилл верификации — 32 механизма

## Архитектура

```
verification/
├── SKILL.md                          ← ТЫ ЗДЕСЬ (Project Instructions — системный промт)
├── references/
│   ├── mechanisms_factual.md         ← №1,2,6,7: факты, хронология, противоречия
│   ├── mechanisms_logical.md         ← №10-17,30: логика, каузальность, стресс-тест
│   ├── mechanisms_source.md          ← №18,19,28: источники, эпистемический статус
│   ├── mechanisms_numerical.md       ← №3,4,20,23: расчёты, границы, метаморфизм
│   ├── mechanisms_document.md        ← №5,8,9,21,22,24-26,29,32: формат, diff, ссылки
│   ├── mechanisms_audience.md        ← №27,31: аудитория, адресат
│   ├── presets_matrix.md             ← М1-М5, П1-П14, матрица, комбинации
│   └── verification_self_mode.md     ← workflow самопроверки для Claude.ai
├── scripts/                          ← опционально, для computer use
│   ├── requirements.txt              ← зависимости: python-docx, openpyxl
│   ├── auto_verify.py                ← оркестратор проверок
│   ├── verify_docx_format.py         ← формат docx по стандарту #6
│   ├── verify_sums.py                ← сверка сумм/итогов в xlsx
│   ├── verify_references.py          ← битые ссылки в docx
│   ├── verify_cross_file.py          ← кросс-файловая согласованность
│   ├── verify_numbering.py           ← нумерация таблиц, диаграмм, приложений
│   ├── verify_dates.py               ← хронология, валидация дат
│   ├── verify_pptx_format.py         ← формат слайдов (16:9, шрифты)
│   ├── verify_pptx_html_sync.py      ← согласованность pptx↔html
│   ├── diff_versions.py              ← diff между версиями файлов
│   └── generate_report.py            ← генерация итогового отчёта
└── INSTALL.md                        ← инструкция по установке в Claude.ai проект
```

**SKILL.md** = Project Instructions (загружается целиком в контекст).
**references/** = Project Knowledge (доступны через `project_knowledge_search`).
**scripts/** = опционально, для computer use.

| Конкретные механизмы по номерам | Нужный `mechanisms_*` по карте ниже |

## Основной workflow

### Этап 0. Автодетекция (до начала работы)

При получении задачи определи:
1. **Тип задачи**: аналитика / расчёт / презентация / документ / перевод / исследование / мероприятие
2. **Входные файлы**: количество, форматы
3. **Выходные файлы**: количество, форматы
4. **Адресат**: кто будет читать (если указан или следует из контекста)

### Этап 1. Автоподбор пресета

На основе Этапа 0 подбери:
- **Рекомендуемый пресет** (П1–П14) или комбинацию — ищи `presets_matrix` через project_knowledge_search
- **Уровень малой верификации** (М1–М5) — включается автоматически
- **Скриптовые проверки** — какие скрипты запустить (если computer use доступен)
- **LLM-проверки** — какие механизмы проверить

### Этап 2. 🔒 КОНТРОЛЬНАЯ ТОЧКА (обязательная)

**НИКОГДА не пропускай этот этап.** Покажи пользователю план и дождись одобрения.

Формат сообщения:
```
📋 План верификации

Рекомендуемый пресет: [название]
Обоснование: [почему именно этот]

Скриптовые проверки (если доступен computer use):
  ☑ [список скриптов]

LLM-проверки:
  ☑ [список механизмов]
```

Используй ask_user_input с вариантами:
- «Всё ок — начинай»
- «Добавить механизмы — выберу»
- «Убрать лишнее — выберу»
- «Сменить пресет»
- «Без верификации»

**Правила:**
- Если пользователь говорит «без верификации» или «нет» → не проводи
- Если ответ неясен → переспроси
- Одного подтверждения достаточно, не переспрашивай «вы уверены?»

### Этап 3. Выполнение основной задачи

Создай документ/файл/анализ. Верификация проводится ПОСЛЕ завершения, не во время.

### Этап 4. Запуск верификации

**Скриптовые проверки** (если computer use доступен):
```bash
python scripts/auto_verify.py \
  --files result.docx source1.xlsx \
  --preset П1 \
  --scripts format,sums,references \
  --output verification_report.md
```

**LLM-проверки** — выполни одобренные механизмы по процедурам из reference-файлов.
Ищи процедуры через `project_knowledge_search` по номеру механизма.

**Если computer use НЕ доступен** — выполни все проверки как LLM, используя fallback-инструкции.
Ищи `verification_self_mode` для workflow самопроверки.

### Этап 5. Отчёт верификации

```markdown
### 📊 Результаты верификации

**Пресет:** [название]
**Механизмы:** [перечень №]
**Режим:** self (Claude.ai)

#### Скриптовые проверки (если запускались)
| Скрипт | Результат | Детали |
|--------|-----------|--------|

#### LLM-проверки
| № | Механизм | Результат | Детали |
|---|----------|-----------|--------|

**Уровень уверенности:** [высокий/средний/низкий] — [обоснование]
**Рекомендации:** [если есть ⚠️ или ❌ — что исправить]
```

## Быстрые сценарии

### Аналитическая записка по нескольким источникам
→ Пресет: П1 «Аналитик» + М2 «Расширенная»
→ Скрипты: format, references, dates
→ LLM: обратная логика, стресс-тест, полнота, спор «за/против»
→ Контрольная точка → работа → верификация → отчёт

### Презентация pptx + html
→ Пресет: М4 «Презентационная» + элементы П7 «Спикер»
→ Скрипты: pptx_format, pptx_html_sync
→ LLM: адресат, полнота, декомпозиция
→ Контрольная точка → работа → верификация → отчёт

### Бюджет/KPI в xlsx + обоснование в docx
→ Пресет: П3 «Бухгалтер» + М2 «Расширенная»
→ Скрипты: sums, cross_file, format, references
→ LLM: двойной расчёт, границы, метаморфическое тестирование
→ Контрольная точка → работа → верификация → отчёт

### Мероприятие/сценарий + таблица спикеров
→ Пресет: П8 «Организатор» + М3 «Максимальная малая»
→ Скрипты: cross_file, dates, numbering
→ LLM: хронология (тайминги), полнота, противоречия
→ Контрольная точка → работа → верификация → отчёт

### Аудит чужого документа
→ Пресет: П13 «Аудитор»
→ Скрипты: format, references, sums, dates
→ LLM: происхождение, стресс-тест, парадоксы, допущения
→ Контрольная точка → работа → верификация → отчёт
