## №3 — Сверка сумм {#3-сверка-сумм}

### Суть
Сумма частей равна целому. Итоги по строкам сходятся с итогами по столбцам. Процентные доли дают 100%.

### Когда применять
- Бюджеты, ФОТ, финансовые таблицы
- KPI-таблицы с итогами
- Любые документы с числовыми итогами

### Маркировка: скрипт + LLM

### Пошаговая процедура

**Шаг 1. Найди все итоговые значения**
Ищи слова: «итого», «всего», «сумма», «total», знак Σ.

**Шаг 2. Пересчитай каждый итог**
Сложи все слагаемые вручную (или скриптом). Сравни с указанным итогом.

**Шаг 3. Проверь процентные распределения**
Если есть «доли» — они должны давать 100% (±0.5% на округление).

**Шаг 4. Кросс-проверка строки↔столбцы**
В двумерных таблицах: итог последней строки = итог последнего столбца.

**Шаг 5. Фиксация**
```
Итогов проверено: [N]
Совпадают: [N]
Расхождения: [N] — [ячейка, ожидалось, получено, разница]
```

### Шаблон фиксации
```
| Ячейка | Ожидалось | Получено | Разница | Статус |
|--------|-----------|----------|---------|--------|
```

### Пример — KPI_FOT_v3 (ЦПИКП vs Департамент)

> **ЦПИКП (23 чел.):**
> | Позиция | Кол-во | Оклад×12 | Переменная | Итого |
> |---------|--------|----------|------------|-------|
> | Руководитель | 1 | 4 800 000 | 5 975 370 | 10 775 370 |
> | Зам. руководителя | 1 | 4 200 000 | 3 925 483 | 8 125 483 |
> | ... | ... | ... | ... | ... |
> | **ИТОГО** | **23** | **59 292 000** | **68 946 000** | **128 238 000** |
>
> Проверка: 59 292 000 + 68 946 000 = 128 238 000 ✅
> Доля переменной: 68 946 000 / 128 238 000 = 53.8% (в документе указано 69%) ❌
>
> → Расхождение: в документе процент считается от другой базы. Уточнить методику.

### Fallback для Claude.ai
Если computer use недоступен: найти все строки с «итого»/«всего»/«сумма», вручную пересчитать каждый итог, проверить процентные доли. Особое внимание — таблицам с двумя направлениями суммирования.

## №4 — Проверка границ {#4-проверка-границ}

### Суть
Числовые значения в допустимых диапазонах. Процент ≤100%, бюджет ≥0, дата не в будущем (если речь о прошлом), количество людей — целое число.

### Когда применять
- Финансовые таблицы
- KPI-модели
- Любые документы с числовыми данными

### Маркировка: скрипт + LLM

### Пошаговая процедура

**Шаг 1. Определи типы данных**
Для каждого числового поля определи допустимый диапазон:

| Тип | Мин | Макс | Дополнительно |
|-----|-----|------|---------------|
| Процент | 0 | 100 (или 150 для KPI) | Не может быть отрицательным |
| Бюджет (руб.) | 0 | — | Порядок величин: тысячи? миллионы? миллиарды? |
| Количество людей | 1 | — | Целое число |
| Дата (прошлое) | — | сегодня | Не в будущем |
| KPI (%) | 0 | 150–200 | Зависит от системы |
| Оклад (мес.) | МРОТ | — | Проверить на «здравый смысл» |

**Шаг 2. Проверь каждое значение**
Особое внимание на:
- Отрицательные значения там, где их быть не может
- Нули в обязательных полях
- Выбросы (значение на порядок отличается от соседних)

**Шаг 3. Проверь порядок величин**
Частая ошибка — перепутаны тысячи и миллионы, или потерян разряд.

**Шаг 4. Фиксация**
```
Значений проверено: [N]
В допустимых границах: [N]
Выбросы/нарушения: [N] — [значение, ожидаемый диапазон, где найдено]
```

### Шаблон фиксации
```
| Значение | Ожидаемый диапазон | Где найдено | Статус |
|----------|-------------------|-------------|--------|
```

### Пример — KPI-вестинг, пересчёт ФОТ

> Коэффициент масштабирования переменной Департамента: 2.971x
> Граница здравого смысла: коэффициент >3x означает, что переменная часть превышает фикс в 3 раза
> → ⚠️ Значение на грани. При KPI=150% общий доход вырастет в 2.971×1.5 = 4.46 раза от фикса.
> → Рекомендация: проверить, реалистичен ли такой уровень мотивации для госсектора.

### Fallback для Claude.ai
Если computer use недоступен: определить типы числовых полей, проверить диапазоны вручную, обратить внимание на порядок величин и «здравый смысл» значений.

## №20 — Двойной расчёт {#20-двойной-расчёт}

### Суть
Каждое ключевое вычисление проводится двумя независимыми методами. Если результаты совпадают — уверенность высокая. Если нет — маркировка расхождения.

### Когда применять
- Финансовые расчёты (ФОТ, бюджеты, прогнозы)
- KPI-моделирование
- Любые вычисления, на основе которых принимаются решения

### Маркировка: LLM-only

### Пошаговая процедура

**Шаг 1. Определи ключевые вычисления**
Не всё нужно считать дважды. Только:
- Итоговые суммы, от которых зависят решения
- Расчёты с несколькими шагами (формула в формуле)
- Прогнозные значения

**Шаг 2. Метод 1 — прямой расчёт**
Вычисли стандартным способом (как в исходном документе).

**Шаг 3. Метод 2 — альтернативный путь**

| Исходный метод | Альтернативный метод |
|----------------|---------------------|
| Сумма по строкам | Сумма по столбцам |
| Формула | Ручной пересчёт по шагам |
| Абсолютные числа | Проверка через проценты |
| Excel-формула | Python-скрипт |
| Снизу вверх (от позиций к итогу) | Сверху вниз (от бюджета к позициям) |

**Шаг 4. Сравнение**
```
Расчёт: [описание]
Метод 1: [значение] (способ: [какой])
Метод 2: [значение] (способ: [какой])
Расхождение: [абсолютное и в %]
Статус: ✅ совпадает / ⚠️ расхождение в пределах округления / ❌ расхождение
```

### Шаблон фиксации
```
| Расчёт | Метод 1 | Метод 2 | Расхождение | Статус |
|--------|---------|---------|-------------|--------|
```

### Пример — ФОТ ЦПИКП при пересчёте 1.5x

> **Задача:** Департамент (30 чел.) должен получать ФОТ = 1.5 × ЦПИКП (23 чел.)
>
> **Метод 1 (снизу вверх):** пересчитать каждую позицию Департамента:
> Сумма окладов×12 = 76 986 000 (фикс)
> Новая переменная = 115 371 000
> Итого: 76 986 000 + 115 371 000 = 192 357 000
>
> **Метод 2 (сверху вниз):** 128 238 000 × 1.5 = 192 357 000
>
> **Расхождение:** 0 руб. ✅

## №23 — Метаморфическое тестирование {#23-метаморфическое-тестирование}

### Суть
Изменение входных данных должно приводить к предсказуемому изменению выхода. Если увеличили бюджет на 10%, а показатели не изменились — что-то не так с моделью.

### Когда применять
- Финансовые модели с формулами
- Прогнозы и сценарии
- Excel-файлы с зависимыми ячейками

### Маркировка: LLM-only

### Пошаговая процедура

**Шаг 1. Определи входные переменные**
Какие данные являются «входами» модели?

**Шаг 2. Сформулируй ожидаемые метаморфозы**

| Если изменить... | То ожидается... |
|-------------------|-----------------|
| KPI с 100% до 150% | ФОТ вырастет (пропорционально переменной) |
| Штат с 23 до 30 | ФОТ вырастет (не линейно — зависит от позиций) |
| Ставку НДФЛ с 13% до 15% | Чистый доход упадёт |
| Количество мероприятий с 51 до 54 | Бюджет программы вырастет |

**Шаг 3. Проведи мутацию**
Измени один входной параметр, пересчитай результат.

**Шаг 4. Проверь соответствие ожиданию**
- Направление изменения правильное? (вырос/упал)
- Масштаб изменения реалистичный? (не в 100 раз)
- Нет ли «замороженных» ячеек, которые не реагируют на изменение?

**Шаг 5. Фиксация**
```
Мутация: [что изменено, было→стало]
Ожидание: [что должно произойти]
Результат: [что произошло]
Статус: ✅ ожидаемо / ❌ неожиданно — [объяснение]
```

### Шаблон фиксации
```
| Мутация | Ожидание | Результат | Статус |
|---------|----------|-----------|--------|
```

### Пример — KPI-вестинг

> **Мутация:** KPI Департамента 100% → 0%
> **Ожидание:** ФОТ упадёт до фиксированной части (76 986 000 руб.)
> **Результат:** 76 986 000 руб. ✅
>
> **Мутация:** KPI Департамента 100% → 150%
> **Ожидание:** ФОТ вырастет на 50% от переменной × коэфф.
> **Результат:** 76 986 000 + 115 371 000 × 1.5 = 250 042 500 руб.
> **Проверка здравого смысла:** средний доход/чел. = 8 334 750 руб./год ≈ 694 563 руб./мес.
> → ⚠️ Для госсектора высоковато. Но для коммерческого подразделения при привлечении инвестиций — допустимо.
