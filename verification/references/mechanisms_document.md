## №5 — Проверка формата документа {#5}

### Суть
Документ соответствует заданным требованиям к формату (стандарт #6 для docx).

### Когда применять
- Любые создаваемые docx-документы
- Документы с требованиями к оформлению

### Маркировка: скрипт

### Чеклист для docx (стандарт #6)

| Параметр | Стандарт | Как проверить |
|----------|----------|---------------|
| Ориентация | A4 книжная | Скрипт: section.orientation |
| Поля верх/низ | 2 см | Скрипт: top_margin/bottom_margin |
| Поле лево | 3 см | Скрипт: left_margin |
| Поле право | 1.5 см | Скрипт: right_margin |
| Шрифт основной | Times New Roman 14pt | Скрипт: font.name, font.size |
| Выравнивание | По ширине | Скрипт: alignment |
| Красная строка | 1.5 см | Скрипт: first_line_indent |
| Межстрочный | 1.15 | Скрипт: line_spacing |
| После абзаца | 8pt | Скрипт: space_after |
| H1 | 22pt жирный #0070C0 | Скрипт |
| H2 | 18pt жирный #0070C0 | Скрипт |
| H3 | 16pt жирный #0070C0 | Скрипт |
| Заголовки выравнивание | По левому краю | Скрипт |
| Таблицы шрифт | 12pt | Скрипт |
| Таблицы межстрочный | 1.0 | Скрипт |
| Таблицы после абзаца | 2pt | Скрипт |
| Верхний колонтитул | 9pt курсив #999999 справа | Скрипт |
| Нижний колонтитул | 12pt нумерация по центру | Скрипт |

### Fallback для Claude.ai
Если computer use недоступен: проверить визуально шрифт (Times New Roman 14pt), поля (3 см слева, 1.5 см справа, 2 см верх/низ), заголовки (22/18/16pt, синие #0070C0), красную строку (1.5 см), межстрочный интервал (1.15).

## №8 — Формат слайдов {#8}

### Суть
Презентация соответствует стандартам оформления.

### Когда применять
- Создание pptx-презентаций
- Проверка чужих презентаций

### Маркировка: скрипт

### Чеклист для pptx

| Параметр | Стандарт |
|----------|----------|
| Соотношение сторон | 16:9 |
| Шрифт минимальный | ≥14pt для текста, ≥18pt для заголовков |
| Количество текста | ≤6 строк на слайд, ≤8 слов на строку |
| Спикер-ноты | Присутствуют на каждом содержательном слайде |
| Единообразие | Шрифт, цвета, отступы — одинаковые на всех слайдах |
| Контрастность | Текст читаем на фоне |

### Fallback для Claude.ai
Если computer use недоступен: проверить соотношение 16:9, минимальный размер шрифтов (≥14pt текст, ≥18pt заголовки), количество текста на слайде (≤6 строк), наличие спикер-нот.

## №9 — Согласованность pptx/html {#9}

### Суть
При создании презентации в двух форматах (pptx + html) содержание должно быть идентичным.

### Когда применять
- Создание презентаций в двух форматах одновременно

### Маркировка: скрипт

### Процедура
1. Извлеки текст из каждого слайда pptx
2. Извлеки текст из каждого «слайда» html
3. Сравни попарно: слайд N pptx ↔ секция N html
4. Проверь: количество слайдов совпадает? Порядок тот же?

### Шаблон фиксации
```
| Слайд/секция | pptx | html | Совпадение | Статус |
|--------------|------|------|------------|--------|
```

### Fallback для Claude.ai
Если computer use недоступен: сравнить слайды попарно — слайд N в pptx ↔ секция N в html. Проверить количество, порядок и содержание.

## №21 — Сверка вход-выход {#21}

### Суть
Исходные данные корректно отражены в результате. Ничего не потеряно, ничего не добавлено «из воздуха».

### Когда применять
- Сведение нескольких файлов в один
- Создание документа по нескольким источникам

### Маркировка: LLM-only

### Процедура
1. Составь реестр входных данных (все факты из источников)
2. Проверь: каждый входной факт, который должен быть в результате, — присутствует?
3. Проверь: нет ли в результате данных, которых не было во входных?
4. Если есть «новые» данные — откуда они? (генерация Claude? логический вывод? другой источник?)

### Пример — сведение таблиц спикеров Конгресса
> **Вход:** Файл 1 (10 спикеров сессии 1.8) + Файл 2 (17 участников из файла конгресса)
> **Выход:** Объединённая таблица (27 участников)
>
> Проверка: 10 + 17 = 27 ✅
> Проверка ФИО: все 10 из файла 1 присутствуют ✅
> Проверка ФИО: все 17 из файла 2 присутствуют ✅
> «Лишние»: 0 ✅

### Шаблон фиксации
```
| Элемент | Источник | В результате | Статус |
|---------|----------|--------------|--------|
```

## №22 — Согласованность файлов {#22}

### Суть
Данные между несколькими файлами проекта не противоречат друг другу.

### Когда применять
- Многофайловые проекты (docx + xlsx + pptx)
- Пакеты документов

### Маркировка: скрипт + LLM

### Процедура
1. Определи «общие точки» — данные, которые фигурируют в нескольких файлах
2. Сверь значение в каждом файле
3. При расхождении — укажи, какой файл «главный» (источник истины)

### Пример — ЦПИКП (docx + xlsx + pptx)
> | Показатель | docx (обоснование) | xlsx (KPI_FOT_v3) | pptx (презентация) | Статус |
> |------------|-------------------|-------------------|--------------------|--------|
> | Штат ЦПИКП | 23 чел. | 23 чел. | 22 чел. | ❌ pptx |
> | ФОТ при 100% | 128 238 000 | 128 238 000 | 128.2 млн | ✅ (округление) |
> | Доля переменной | 69% | 53.8% | 69% | ⚠️ разная база расчёта |

### Шаблон фиксации
```
| Показатель | Файл 1 | Файл 2 | Файл 3 | Статус |
|------------|--------|--------|--------|--------|
```

### Fallback для Claude.ai
Если computer use недоступен: составить реестр общих точек данных между файлами, сверить каждую попарно вручную. При расхождении — указать источник истины.

## №24 — Diff было/стало {#24}

### Суть
Зафиксированы и проверены все изменения между версиями документа.

### Когда применять
- Многоитерационная работа
- Редактирование документов

### Маркировка: скрипт

### Процедура
1. Сравни две версии (текст, данные, структуру)
2. Составь таблицу изменений: что было → что стало
3. Проверь: каждое изменение — намеренное? Нет ли случайных потерь?

### Шаблон фиксации
```
| Место | Было | Стало | Намеренно | Статус |
|-------|------|-------|-----------|--------|
```

### Fallback для Claude.ai
Если computer use недоступен: сравнить два текста, выписать все отличия вручную, проверить намеренность каждого изменения.

## №25 — Защита от регрессии {#25}

### Суть
Ранее исправленные ошибки не вернулись в новой версии. Актуально при многоитерационной работе.

### Когда применять
- Доработка документа после замечаний
- Версия N+1 после правок

### Маркировка: LLM-only

### Процедура
1. Возьми список ошибок, найденных на предыдущей итерации
2. Проверь каждую: исправлена ли? Не вернулась ли?
3. Если вернулась — ❌ регрессия

### Пример
> Итерация 1: «Приложение Г не приложено» → исправлено
> Итерация 2: добавлена новая таблица → Приложение Г снова пропало (нумерация сбилась)
> → ❌ Регрессия

### Шаблон фиксации
```
| Ошибка (итерация N-1) | Статус в итерации N | Комментарий |
|------------------------|---------------------|-------------|
```

## №26 — Дрейф смысла {#26}

### Суть
Исходный смысл не искажён при переработке, редактировании, пересказе.

### Когда применять
- Переводы
- Сжатие длинных текстов
- Редактирование формулировок

### Маркировка: LLM-only

### Процедура
1. Сравни ключевые тезисы в исходнике и в результате
2. Для каждого: сохранён ли смысл? Не добавлены ли оттенки, которых не было?
3. Особое внимание: при переводе, при сжатии длинного текста, при «причёсывании» формулировок

### Пример — перевод тезисов
> **Оригинал (рус):** «Необходимо рассмотреть возможность увеличения финансирования»
> **Перевод (англ):** «Funding must be increased»
> → ❌ Дрейф: «рассмотреть возможность» ≠ «must be increased». Модальность изменена с рекомендательной на императивную.

### Шаблон фиксации
```
| Тезис (оригинал) | Тезис (результат) | Смысл сохранён | Комментарий |
|-------------------|--------------------|----------------|-------------|
```

## №29 — Кросс-модальная проверка {#29}

### Суть
Согласованность между текстом, таблицами, графиками и визуалами в одном документе.

### Когда применять
- Документы с таблицами и графиками
- Презентации с диаграммами

### Маркировка: LLM-only

### Процедура
1. Найди все места, где текст ссылается на таблицу/график
2. Проверь: числа в тексте совпадают с числами в таблице/графике?
3. Проверь: подписи к графикам соответствуют содержанию?
4. Проверь: легенды графиков корректны?

### Пример — Т-Банк презентация
> Текст слайда 5: «Структура выручки: доля DIGITAL — 34%»
> Donut-диаграмма на этом же слайде: сегмент DIGITAL = 34% ✅
> Текст слайда 11: «Целевая аллокация: ОФЗ 25,7%»
> Stacked bar на слайде: ОФЗ сегмент = 25,7% ✅

### Шаблон фиксации
```
| Текст | Таблица/График | Совпадение | Статус |
|-------|----------------|------------|--------|
```

## №32 — Ссылочная целостность {#32}

### Суть
Все внутренние ссылки ведут к существующим элементам с корректным содержанием.

### Когда применять
- Длинные документы с внутренними ссылками
- Документы с приложениями

### Маркировка: скрипт + LLM

### Процедура

**Шаг 1. Сбор ссылок**
Паттерны: «см. п. X», «Таблица N», «Приложение X», «на слайде N», «подробнее в файле X», «как показано выше/ниже».

**Шаг 2. Проверка существования**
Каждая ссылка → существует ли цель?

**Шаг 3. Проверка содержания**
Цель существует — но содержит ли то, что обещает ссылка?

**Шаг 4. Карта ссылок**
```
Ссылка → Цель → Статус
«см. Таблицу 2» → Табл. 2, стр. 5 → ✅
«Приложение В» → — → ❌ не приложено
«на слайде 8» → Слайд 8 → ⚠️ тема другая
```

### Шаблон фиксации
```
| Ссылка | Цель | Существует | Содержание верно | Статус |
|--------|------|------------|------------------|--------|
```

### Fallback для Claude.ai
Если computer use недоступен: найти все ссылки по паттернам («см.», «Таблица N», «Приложение X»), для каждой проверить существование цели и корректность содержания.
