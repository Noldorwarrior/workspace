## Режим self — описание {#режим-self}

В Claude.ai доступен только режим **self** — Claude проверяет свою работу сам, без субагентов.

**Ключевой риск:** confirmation bias — склонность подтверждать собственную правоту. Claude склонен считать свою работу корректной, потому что он сам её написал.

**Когда используется:**
- Любые задачи в Claude.ai (единственный доступный режим)
- Простые задачи в Claude Code (М1–М3)

**Ограничения:**
- Нет независимого верификатора
- Нет параллельных проверок
- Нет оркестрации

---

## Приёмы против confirmation bias {#приёмы-против-confirmation-bias}

### 1. Перечитай документ «глазами критика»
Представь, что ты НЕ автор. Ты — независимый рецензент, которому заплатили за поиск ошибок. Твоя задача — найти проблемы, а не подтвердить качество.

### 2. Проверяй по процедуре, не по памяти
Не полагайся на то, что «помнишь, что проверял». Открой reference-файл через `project_knowledge_search` и следуй шагам процедуры буквально.

### 3. Не пропускай шаги процедуры
Даже если «очевидно, что всё ок» — выполни каждый шаг. «Очевидно ок» — главный маркер confirmation bias.

### 4. Правило нулевых ошибок
Если нашёл 0 ошибок — это подозрительно. Перепроверь самый рискованный механизм (тот, где ошибка наиболее вероятна или наиболее опасна).

### 5. Сначала закончи — потом проверяй
Не проверяй документ по мере написания. Закончи → отдели этап создания от этапа верификации → проверяй как чужой документ.

## Fallback-инструкции для скриптовых механизмов {#fallback-инструкции}

Если computer use недоступен, все скриптовые проверки выполняются LLM вручную:

| Скрипт | LLM-fallback (что делать вручную) |
|--------|-----------------------------------|
| verify_docx_format.py | Проверить шрифт (Times New Roman 14pt), поля (3/1.5/2/2 см), заголовки (22/18/16pt синие), красную строку (1.5 см), межстрочный (1.15). Пройти по чеклисту стандарта #6 из mechanisms_document.md. |
| verify_sums.py | Найти все строки с «итого»/«всего»/«сумма». Пересчитать каждый итог вручную. Проверить: процентные доли дают 100%? Итоги строк = итоги столбцов? |
| verify_references.py | Найти все паттерны ссылок: «см.», «Таблица N», «Приложение X», «п. N», «на слайде N». Для каждой ссылки проверить: существует ли цель? Содержит ли то, что обещает ссылка? |
| verify_cross_file.py | Составить реестр общих точек данных между файлами (числа, ФИО, даты). Сверить каждую точку попарно между файлами. При расхождении — указать источник истины. |
| verify_numbering.py | Пройти по всем нумерованным элементам (Таблица 1, 2, 3...; Рисунок 1, 2...; Приложение А, Б...). Проверить: нет ли дубликатов? Нет ли пропусков? Последовательность корректна? |
| verify_dates.py | Выписать все даты из документа. Построить временную шкалу. Проверить: хронологический порядок корректен? Нет ли дат в будущем (без прогнозного контекста)? Реалистичны ли интервалы? |
| verify_pptx_format.py | Проверить: соотношение сторон 16:9? Минимальный шрифт ≥14pt? Не более 6 строк текста на слайд? Есть ли спикер-ноты? Единообразие шрифтов и цветов? |
| verify_pptx_html_sync.py | Сравнить слайды попарно: слайд N в pptx ↔ секция N в html. Проверить: количество совпадает? Содержание каждого слайда/секции идентично? |
| diff_versions.py | Сравнить два текста, выписать все отличия: что добавлено, что удалено, что изменено. Проверить намеренность каждого изменения. |
| generate_report.py | Сформировать отчёт в формате markdown по шаблону из SKILL.md. Заполнить таблицы скриптовых и LLM-проверок, указать уровень уверенности и рекомендации. |

## Workflow самопроверки в Claude.ai {#workflow-самопроверки}

```
Получил задачу от пользователя
↓
Этап 0: Определил тип задачи, файлы, адресата
↓
Этап 1: Подобрал пресет → нашёл presets_matrix через project_knowledge_search
↓
Этап 2: 🔒 КОНТРОЛЬНАЯ ТОЧКА → показал план через ask_user_input → ждал одобрения
↓
Этап 3: Создал документ / выполнил задачу
↓
СТОП. Не отправляй пользователю.
↓
Открой presets_matrix.md → найди свой пресет → список механизмов
↓
Для каждого механизма:
  → Найди нужный mechanisms_*.md через project_knowledge_search
  → Следуй пошаговой процедуре
  → Если механизм скриптовый и computer use доступен → запусти скрипт
  → Если computer use НЕ доступен → используй LLM-fallback из таблицы выше
↓
Сформируй блок «Результаты верификации» по шаблону из SKILL.md
↓
Отправь пользователю: документ + результаты верификации
```

### Важные правила workflow

1. **Верификация ПОСЛЕ создания, не во время** — сначала закончи документ, потом проверяй
2. **Следуй процедуре** — не проверяй «по памяти», открой reference-файл
3. **Не пропускай контрольную точку** — даже если пользователь «торопит»
4. **0 ошибок = подозрительно** — перепроверь самый рискованный механизм
5. **Показывай детали** — не «Всё ок ✅», а конкретные результаты по каждому механизму

## Антипаттерны {#антипаттерны}

### ❌ Верификация во время создания
Не проверяй документ ПО МЕРЕ написания. Сначала создай полностью, потом проверяй. Иначе verification bias — ты «знаешь», что всё правильно, потому что сам только что написал.

### ❌ Формальная отписка
```
Верификация проведена. Всё ок. ✅
```
Это не верификация. Нужны конкретные результаты по каждому механизму с деталями.

### ❌ Пропуск контрольной точки
НИКОГДА не запускай верификацию без одобрения пользователя. Даже если «очевидно» нужна. Даже если в инструкциях написано «всегда верифицировать».

### ❌ Автоматическое исправление без уведомления
Если верификация нашла ошибку — ПОКАЖИ пользователю, а не исправляй молча. Пользователь решает, исправлять ли и как.

### ❌ Повторная верификация без изменений
Если исправил ошибку → перезапусти ТОЛЬКО те механизмы, которые нашли проблему, а не все 32 заново. Используй пресет П14 «Итератор» для повторных проверок.

## Примечание для Claude Code {#примечание-для-claude-code}

> При использовании в Claude Code доступны расширенные режимы:
> - **subagent** — независимый субагент-верификатор (не видел процесс создания, проверяет «с чистого листа»)
> - **parallel** — 2–3 параллельных субагента по специализациям (Логик, Фактчекер, Редактор)
> - **orchestrator** — мастер-оркестратор для П5 «Максимум» (полная оркестрация всех 32 механизмов)
>
> Документация расширенных режимов: см. исходный agent_patterns.md.
>
> Автовыбор режима в Claude Code:
> - claude.ai → self (единственный вариант)
> - Claude Code + М1–М5 только → self + скрипты
> - Claude Code + П1–П4,П6–П11 → subagent
> - Claude Code + П12–П14 → parallel (2–3 субагента)
> - Claude Code + П5 → orchestrator
> - Claude Code + комбинация 3+ пресетов → parallel
