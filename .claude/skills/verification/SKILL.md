## Основной workflow

### Этап 0. Автодетекция (до начала работы)

При получении задачи определи:
1. **Тип задачи**: аналитика / расчёт / презентация / документ / перевод / исследование / мероприятие
2. **Входные файлы**: количество, форматы
3. **Выходные файлы**: количество, форматы
4. **Адресат**: кто будет читать (если указан или следует из контекста)

**Fallback:** если тип задачи не определяется однозначно — спроси пользователя: «Какой тип задачи ближе: аналитика, расчёт, презентация, документ, перевод, исследование, мероприятие?». Если пользователь не уточняет — используй М1 «Базовая» как минимальный уровень.

### Этап 1. Автоподбор пресета

На основе Этапа 0 подбери:
- **Рекомендуемый пресет** (П1–П14) или комбинацию
- **Уровень малой верификации** (М1–М5) — включается автоматически
- **Скриптовые проверки** — какие скрипты запустить
- **LLM-проверки** — какие механизмы проверить агентом
- **Режим агентов** (только для Claude Code):
  - `self` — Claude проверяет сам (claude.ai; в Claude Code — для малых пресетов М1–М5)
  - `subagent` — субагент-верификатор (Claude Code, стандарт)
  - `parallel` — параллельные субагенты (Claude Code, сложные задачи)
  - `orchestrator` — мастер-оркестратор (Claude Code, П5 Максимум)

### Этап 2. 🔒 КОНТРОЛЬНАЯ ТОЧКА (обязательная)

**НИКОГДА не пропускай этот этап.** Покажи пользователю план и дождись одобрения.

Формат сообщения пользователю:
```
📋 План верификации

Рекомендуемый пресет: [название] 
Обоснование: [почему именно этот]

Скриптовые проверки (автоматические):
  ☑ [список скриптов с описанием]

LLM-проверки (агентные):
  ☑ [список механизмов]

Режим: [self / subagent / parallel / orchestrator]

→ Что добавить или убрать?
→ Какие способы одобряешь?
→ Подтверди для старта.
```

В claude.ai — используй ask_user_input с вариантами:
- «Всё ок — начинай»
- «Добавить механизмы — выберу»
- «Убрать лишнее — выберу»  
- «Сменить пресет»
- «Без верификации»

В Claude Code — задай вопрос в чате, дождись ответа.

**Правило:** если пользователь говорит «без верификации» или «нет» — не проводи. 
Если молчит или неясно — переспроси, не начинай без подтверждения.

### Этап 3. Выполнение основной задачи

Создай документ/файл/анализ. Верификация проводится ПОСЛЕ завершения, не во время.

**Критерии готовности к верификации:**
- Документ полностью создан (все секции заполнены, нет плейсхолдеров вида `[TODO]`)
- Все выходные файлы сохранены в финальных форматах (docx, xlsx, pptx)
- Если задача многофайловая — все файлы готовы одновременно (для кросс-файловой проверки)

### Этап 4. Запуск верификации

**Скриптовые проверки** — запусти одобренные скрипты:
```bash
python scripts/auto_verify.py \
  --files result.docx source1.xlsx \
  --preset П1 \
  --scripts format,sums,references \
  --output verification_report.md
```

**LLM-проверки** — выполни одобренные механизмы по процедурам из reference-файлов.

В Claude Code с субагентами:
```bash
# Субагент-верификатор получает:
# 1. Файл для проверки
# 2. Исходные материалы
# 3. Список механизмов с процедурами
# 4. Формат отчёта
```

### Этап 5. Отчёт верификации

Формат итогового блока:

```markdown
### 📊 Результаты верификации

**Пресет:** [название]  
**Механизмы:** [перечень №]  
**Режим:** [self / subagent / parallel]

#### Скриптовые проверки (автоматические)
| Скрипт | Результат | Детали |
|--------|-----------|--------|
| verify_docx_format.py | ✅ | Формат соответствует стандарту #6 |
| verify_sums.py | ⚠️ | Итог в ячейке D15 не совпадает с суммой D3:D14 |

#### LLM-проверки
| № | Механизм | Результат | Детали |
|---|----------|-----------|--------|
| 12 | Обратная логика | ✅ | Цепочка аргументации корректна |
| 30 | Стресс-тест | 🟡 | Вывод №3 хрупкий — стоит на одной посылке |

**Уровень уверенности:** [высокий/средний/низкий] — [обоснование]
**Рекомендации:** [если есть ⚠️ или ❌ — что исправить]
```

## Быстрые сценарии

### Сценарий: аналитическая записка по нескольким источникам
→ Авто: П1 «Аналитик» + М2 «Расширенная»
→ Скрипты: format, sums, references (из П1 + М2)
→ LLM: обратная логика, стресс-тест, полнота, спор «за/против»
→ Контрольная точка → работа → верификация → отчёт

### Сценарий: презентация pptx + html
→ Авто: М4 «Презентационная» (№1,2,8,9) + из П7: №27 (аудитория), №31 (адресат), №15 (полнота), №13 (декомпозиция)
→ Скрипты: pptx_format, pptx_html_sync, cross_file
→ LLM: моделирование аудитории, проверка адресата, полнота, декомпозиция фактов
→ Контрольная точка → работа → верификация → отчёт

### Сценарий: бюджет/KPI в xlsx + обоснование в docx
→ Авто: П3 «Бухгалтер» + М2 «Расширенная»
→ Скрипты: sums, cross_file, format, references
→ LLM: двойной расчёт, границы, метаморфическое тестирование
→ Контрольная точка → работа → верификация → отчёт

### Сценарий: сценарий мероприятия + таблица спикеров
→ Авто: П8 «Организатор» + М3 «Максимальная малая»
→ Скрипты: cross_file, dates, numbering
→ LLM: хронология (тайминги), полнота, противоречия
→ Контрольная точка → работа → верификация → отчёт

### Сценарий: проверка чужого документа
→ Авто: П13 «Аудитор»
→ Скрипты: format, references, sums, dates
→ LLM: происхождение, стресс-тест, парадоксы, допущения
→ Контрольная точка → работа → верификация → отчёт
