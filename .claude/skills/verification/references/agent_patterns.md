## Режимы работы

| Режим | Когда | Надёжность | Стоимость токенов |
|-------|-------|------------|-------------------|
| `self` | claude.ai, простые задачи, М1–М3 | Базовая — confirmation bias | Низкая |
| `subagent` | Claude Code, стандартные задачи, П1–П4,П6–П11 | Высокая — независимая проверка | Средняя |
| `parallel` | Claude Code, сложные задачи, П12–П14 | Очень высокая — несколько проверяющих | Высокая |
| `orchestrator` | Claude Code, П5 Максимум, критические документы | Максимальная — полный контроль | Очень высокая |

### Автовыбор режима

```
Входные данные:
  - количество файлов
  - тип пресета
  - количество LLM-механизмов
  - среда выполнения (claude.ai / Claude Code)

Правила:
  claude.ai                         → self (единственный вариант)
  Claude Code + М1–М5 только        → self + скрипты
  Claude Code + П1–П4,П6–П11       → subagent
  Claude Code + П12–П14            → parallel (2–3 субагента)
  Claude Code + П5                 → orchestrator
  Claude Code + комбинация 3+ пресетов → parallel
```

## Паттерн: субагент-верификатор

### Принцип
Основной агент (исполнитель) создаёт документ. Субагент (верификатор) получает:
- Готовый документ
- Исходные материалы
- Исходный запрос пользователя
- Одобренный список механизмов
- Процедуры из reference-файлов

Субагент НЕ видел процесс создания — проверяет «с чистого листа».

### Инструкция для субагента

```
Ты — верификатор. Твоя задача — проверить документ [файл] по следующим механизмам:
[список механизмов с номерами]

Для каждого механизма следуй пошаговой процедуре:
[вставить процедуры из reference-файлов]

Исходный запрос пользователя:
[запрос]

Исходные материалы:
[файлы-источники]

Формат отчёта:
| № | Механизм | Результат | Детали |
|---|----------|-----------|--------|

В конце — общий уровень уверенности и рекомендации.

ВАЖНО:
- Ты не участвовал в создании документа. Проверяй как независимый аудитор.
- Не «додумывай» намерения автора. Проверяй то, что написано.
- Если что-то неясно — маркируй как ⚠️, не пропускай молча.
- Если нашёл критическую ошибку — маркируй ❌ и укажи конкретно что исправить.
```

### Порядок вызова в Claude Code

```bash
# 1. Основной агент завершил работу → файл result.docx

# 2. Запуск скриптовых проверок (параллельно с субагентом)
python scripts/auto_verify.py --files result.docx --scripts format,references,dates

# 3. Запуск субагента-верификатора
# В Claude Code субагент вызывается через Task tool или параллельный процесс
# Субагент получает: result.docx + sources/ + verification_instructions.md

# 4. Сбор результатов
python scripts/generate_report.py \
  --script-results script_report.json \
  --agent-results agent_report.md \
  --output final_verification.md
```

## Паттерн: параллельная верификация

### Принцип
Механизмы распределяются между 2–3 субагентами по специализации. Каждый проверяет свою группу параллельно.

### Распределение по специализациям

```
Субагент A — «Логик»:
  №10 скрытые допущения
  №11 парадоксы
  №12 обратная логика
  №16 спор «за/против»
  №17 граф причин-следствий
  №20 двойной расчёт
  №23 метаморфическое тестирование
  №30 стресс-тест выводов

Субагент B — «Фактчекер»:
  №2 выполнение запроса
  №7 противоречия
  №13 декомпозиция фактов
  №14 оценка уверенности
  №15 полнота
  №18 триангуляция
  №19 происхождение
  №28 эпистемический статус

Субагент C — «Редактор»:
  №21 сверка вход-выход
  №25 защита от регрессии
  №26 дрейф смысла
  №27 моделирование аудитории
  №29 кросс-модальная
  №31 проверка адресата
```

Скриптовые проверки (№1 частично, №3,4,5,6,8,9,22,24,32) запускаются параллельно со всеми субагентами через `auto_verify.py`. Механизм №1 (точный перенос) автоматизирован для чисел и ФИО; семантическую проверку контекста делает LLM.

### Сведение результатов

Если субагенты дают противоречивые оценки одного и того же фрагмента:
1. Зафиксировать расхождение
2. Показать пользователю обе оценки
3. Пользователь решает

## Паттерн: оркестратор

### Когда
П5 «Максимум» — все 32 механизма, критически важные документы.

### Архитектура

```
Мастер-оркестратор
  ├── Этап 0: анализ задачи, подготовка чеклиста
  ├── Этап 1: формирование контрольной точки → одобрение пользователя
  ├── Этап 2: распределение задач
  │   ├── auto_verify.py (все скрипты)          ─┐
  │   ├── Субагент A «Логик»                    ├── параллельно
  │   ├── Субагент B «Фактчекер»                │
  │   └── Субагент C «Редактор»                 ─┘
  ├── Этап 3: сбор результатов
  ├── Этап 4: кросс-проверка (оркестратор сам проверяет расхождения)
  └── Этап 5: генерация финального отчёта
```

### Кросс-проверка оркестратором

После сбора результатов от всех субагентов оркестратор:
1. Ищет расхождения между субагентами
2. Ищет расхождения между скриптами и LLM-проверками
3. Проверяет: не пропущен ли механизм (все 32 покрыты?)
4. Формирует сводный отчёт с единой шкалой уверенности

## Контрольная точка: протокол

### Формат предложения пользователю

```
📋 ПЛАН ВЕРИФИКАЦИИ

┌─ Задача ──────────────────────────────────────┐
│ [краткое описание задачи]                     │
│ Входные файлы: [список]                       │
│ Выходные файлы: [список]                      │
│ Адресат: [если известен]                      │
└───────────────────────────────────────────────┘

┌─ Рекомендация ────────────────────────────────┐
│ Пресет: [название] + [малая]                  │
│ Обоснование: [1 предложение]                  │
└───────────────────────────────────────────────┘

┌─ Скриптовые проверки (автоматические) ────────┐
│ ☑ verify_docx_format.py — формат по #6        │
│ ☑ verify_sums.py — сверка сумм                │
│ ☑ verify_references.py — битые ссылки         │
│ ☐ verify_cross_file.py — кросс-файл           │
│ ☐ verify_dates.py — валидация дат             │
│ [☑ = рекомендовано, ☐ = опционально]          │
└───────────────────────────────────────────────┘

┌─ LLM-проверки ────────────────────────────────┐
│ ☑ №12 обратная логика                         │
│ ☑ №30 стресс-тест выводов                     │
│ ☑ №15 полнота                                 │
│ ☐ №16 спор «за/против»                       │
│ [☑ = рекомендовано, ☐ = опционально]          │
└───────────────────────────────────────────────┘

┌─ Режим ───────────────────────────────────────┐
│ subagent (субагент-верификатор)                │
└───────────────────────────────────────────────┘

→ Что добавить или убрать?
→ Какие способы проверки одобряешь?
→ Подтверди для старта.
```

### Обработка ответа пользователя

| Ответ | Действие |
|-------|----------|
| «Всё ок / да / подтверждаю» | Начать работу по плану |
| «Добавь №16 и №17» | Добавить механизмы, показать обновлённый план, ждать повторного подтверждения |
| «Убери скрипты, только LLM» | Убрать скриптовые, показать обновлённый план |
| «Смени на П5» | Пересчитать план для П5, показать, ждать подтверждения |
| «Без верификации / нет» | Не проводить верификацию, приступить к задаче |
| «Сделай сам, доверяю» | Использовать рекомендованный план без изменений |
| Неясный ответ | Переспросить: «Подтверждаешь план или хочешь изменить?» |

### Правило: двойного подтверждения не нужно
Если пользователь один раз подтвердил план — работай. Не переспрашивай «вы уверены?»

## Интеграция со скриптами

### auto_verify.py: интерфейс

```bash
# Минимальный вызов
python scripts/auto_verify.py --files document.docx

# Полный вызов
python scripts/auto_verify.py \
  --files result.docx source.xlsx slides.pptx \
  --preset П1 \
  --scripts format,sums,references,cross_file,dates \
  --output verification_report.json \
  --report-format md
```

## Антипаттерны

### ❌ Верификация во время создания
Не проверяй документ ПО МЕРЕ написания. Сначала создай полностью, потом проверяй. Иначе verification bias — ты «знаешь», что всё правильно, потому что сам только что написал.

### ❌ Формальная отписка
```
Верификация проведена. Всё ок. ✅
```
Это не верификация. Нужны конкретные результаты по каждому механизму.

### ❌ Субагент читает контекст исполнителя
Субагент-верификатор НЕ должен видеть рассуждения исполнителя, его промежуточные черновики, его «мыслительный процесс». Только: файл + источники + ТЗ.

### ❌ Пропуск контрольной точки
НИКОГДА не запускай верификацию без одобрения пользователя. Даже если «очевидно» нужна. Даже если в CLAUDE.md написано «всегда верифицировать».

### ❌ Автоматическое исправление без уведомления
Если верификация нашла ошибку — ПОКАЖИ пользователю, а не исправляй молча. Пользователь решает, исправлять ли и как.

### ❌ Повторная верификация без изменений
Если исправил ошибку → перезапусти ТОЛЬКО те механизмы, которые нашли проблему, а не все 32 заново. Используй пресет П14 «Итератор» для повторных проверок.
