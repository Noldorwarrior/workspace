## Раздел 1. Обзор системы

> **Зачем мне это знать?** Этот раздел даёт общую картину: что такое система верификации, какие проблемы решает и из каких частей состоит. После прочтения вы будете понимать, зачем система нужна и как она устроена на верхнем уровне.

### 1.1. Что это такое

> Представьте отдел контроля качества на заводе. Каждый продукт проходит серию проверок перед отправкой клиенту: визуальный осмотр, измерение размеров, проверка веса, тест на прочность. Система верификации — это такой же «отдел контроля качества», но для документов, которые создаёт или обрабатывает AI-ассистент.

**Система верификации** — это набор инструментов и процедур для проверки результатов работы AI-ассистента (Claude). Она автоматически находит ошибки в документах: неправильно перенесённые цифры, нарушенную логику, пропущенные пункты задания, неверный формат — и выдаёт отчёт с конкретными находками.

Система включает **32 вида проверок** (механизма), объединённых в **19 готовых наборов** (пресетов) для типовых задач. Часть проверок выполняется автоматически Python-скриптами, часть — самим AI, анализирующим текст на смысловом уровне.

### 1.2. Зачем нужна

Даже лучшие AI-модели допускают ошибки. Вот реальные проблемы, с которыми сталкивается каждый:

| Проблема | Пример из жизни | Как верификация решает |
|----------|----------------|----------------------|
| AI «округлил» цифры | Бюджет 1 247 893 руб. превратился в «около 1,2 млн» | **Механизм №1** (точный перенос) посимвольно сравнивает числа в источнике и в итоговом документе |
| Пропущен пункт задания | Попросили 5 разделов — получили 4 | **Механизм №2** (выполнение запроса) разбивает ваш запрос на пункты и проверяет каждый |
| Формат документа нарушен | Шрифт не тот, поля не те, нумерация сбилась | **Скрипт verify_docx_format.py** проверяет 14+ параметров формата по стандарту |
| Смысл «поплыл» при правке | «Повысить эффективность» стало «увеличить результативность» — вроде то же, но юридически нет | **Механизм №26** (дрейф смысла) сравнивает семантику оригинала и итога |
| Даты противоречат друг другу | «Проект стартовал в марте» ... «за первый месяц (апрель)» | **Механизм №6** (хронология) + **скрипт verify_dates.py** проверяют последовательность и реалистичность дат |
| Суммы не сходятся | Итоговая строка 1 500 000, а сумма строк выше — 1 499 000 | **Механизм №3** (сверка сумм) + **скрипт verify_sums.py** пересчитывают все итоги |
| Ссылки «в никуда» | «см. Таблицу 5» — а таблица 5 удалена | **Механизм №32** (ссылочная целостность) + **скрипт verify_references.py** находят битые ссылки |

**Без верификации** эти ошибки находит человек (если повезёт) — после отправки документа руководству. **С верификацией** — система находит их автоматически, до отправки.

### 1.3. Из чего состоит (карта системы)

Система состоит из пяти основных компонентов. Каждый выполняет свою роль:

```
┌──────────────────────────────────────────────────────────────────┐
│                      СИСТЕМА ВЕРИФИКАЦИИ                        │
│                                                                  │
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐           │
│  │ 32 меха-    │   │ 19 пре-     │   │ 11 Python-  │           │
│  │ низма       │   │ сетов       │   │ скриптов    │           │
│  │             │   │             │   │             │           │
│  │ ЧТО         │   │ КАКИЕ       │   │ КАК         │           │
│  │ проверяем   │   │ наборы      │   │ автомати-   │           │
│  │             │   │ применяем   │   │ зируем      │           │
│  └──────┬──────┘   └──────┬──────┘   └──────┬──────┘           │
│         │                 │                 │                   │
│         └─────────────────┼─────────────────┘                   │
│                           ▼                                     │
│              ┌────────────────────────┐                         │
│              │   5-этапный workflow   │                         │
│              │                        │                         │
│              │  КАК устроен процесс   │                         │
│              │  от задачи до отчёта   │                         │
│              └───────────┬────────────┘                         │
│                          ▼                                      │
│              ┌────────────────────────┐                         │
│              │   4 режима агентов     │                         │
│              │                        │                         │
│              │  КТО выполняет:        │                         │
│              │  сам / субагент /      │                         │

└──────────────────────────────────────────────────────────────────┘
```

**Как это работает вместе:**
- Вы описываете задачу — система определяет, какой **пресет** (набор проверок) подходит
- Пресет содержит список **механизмов** — конкретных видов проверки
- Часть механизмов автоматически выполняются **скриптами**, часть — AI-агентом
- Весь процесс управляется **workflow** из 5 этапов
- **Режим агентов** определяет, кто именно проводит проверку (сам AI или его помощники)

### 1.4. Быстрый старт

«Хочу просто включить верификацию — что делать?»

**Шаг 1.** Опишите свою задачу как обычно.

> Пример: «Составь аналитическую записку по трём файлам: отчёт.xlsx, данные.docx, справка.pdf»

**Шаг 2.** Система автоматически предложит план верификации — набор проверок, подходящий для вашей задачи. Вы увидите:
- Рекомендуемый пресет (например, П1 «Аналитик»)
- Список скриптовых проверок (автоматические)
- Список LLM-проверок (смысловые)

**Шаг 3.** Подтвердите план или скорректируйте:
- «Всё ок — начинай» — система работает по плану
- «Добавь проверку логики» — система добавит механизмы
- «Без верификации» — система не будет проверять

**Шаг 4.** Получите отчёт: что проверено, что найдено, что рекомендуется исправить.

> **Важно:** система ВСЕГДА спрашивает вашего одобрения перед запуском проверок. Это называется **контрольная точка** — обязательная пауза, на которой вы решаете, что именно проверять.

## Раздел 2. Глоссарий

> **Зачем мне это знать?** Система верификации использует специфические термины. Этот глоссарий объясняет каждый из них простым языком — с аналогией из повседневной жизни и конкретным примером. Возвращайтесь к нему, когда встретите незнакомое слово в документации.

---

**LLM (Large Language Model)** — большая языковая модель. AI-система, которая понимает и генерирует текст на естественном языке.
_Аналогия:_ очень начитанный собеседник, который прочёл миллиарды текстов и может поддержать разговор на любую тему.
_Пример:_ Claude, ChatGPT, Gemini — всё это LLM.

---

**Агент** — AI-ассистент, который не просто отвечает на вопросы, а выполняет задачи: читает файлы, запускает программы, пишет документы, проверяет результаты.
_Аналогия:_ не просто консультант, который даёт совет, а сотрудник, который берёт задачу и делает работу.
_Пример:_ Claude Code — агент. Он может прочитать ваш файл Excel, написать по нему аналитическую записку в Word и проверить результат.

---

**Верификация** — проверка результата работы на ошибки, неточности и несоответствия. Проводится ПОСЛЕ завершения основной задачи, не во время.
_Аналогия:_ корректура статьи перед публикацией — автор написал, корректор проверяет.
_Пример:_ AI составил бюджет → верификация проверяет: сходятся ли суммы, правильные ли даты, не пропущены ли статьи расходов.

---

**Механизм верификации** — один конкретный вид проверки. В системе их 32 — каждый ищет определённый тип ошибок.
_Аналогия:_ отдельный тест на заводе. Один тест проверяет вес, другой — прочность, третий — цвет. Каждый тест — один механизм.
_Пример:_ механизм №3 «Сверка сумм» — проверяет, что итоговые строки в таблицах равны сумме строк выше.

---

**Скриптовый механизм (🔧)** — проверка, которую выполняет Python-программа. Результат точный и повторяемый — при одних и тех же данных скрипт всегда выдаёт одинаковый ответ.
_Аналогия:_ калькулятор. Вы вводите 2+2 — он всегда отвечает 4. Не «примерно 4» и не «около 4», а ровно 4.
_Пример:_ скрипт verify_sums.py пересчитывает итоги в Excel-таблице и сравнивает с записанными значениями.

---

**LLM-агентный механизм (🧠)** — проверка, которую выполняет AI-модель. Она понимает смысл текста, контекст, логику — то, что скрипт проверить не может.
_Аналогия:_ опытный редактор. Он не просто ищет опечатки — он видит, что аргумент слабый, вывод не следует из посылок, а тон письма не соответствует адресату.
_Пример:_ механизм №12 «Обратная логика» — AI читает вывод документа и проверяет, действительно ли он следует из приведённых аргументов.

---

**Пресет** — готовый набор механизмов, подобранный для типовой задачи. Вместо того чтобы выбирать из 32 проверок вручную, вы выбираете один пресет.
_Аналогия:_ режим стиральной машины. «Хлопок 60°» — это набор настроек: температура, обороты, время. Вы не настраиваете каждый параметр — выбираете режим под свою одежду.
_Пример:_ пресет П3 «Бухгалтер» включает проверку сумм, границ значений, двойной расчёт — всё, что нужно для бюджетных документов.

---

**Малая верификация (М1–М5)** — быстрая базовая проверка. Ловит очевидные ошибки: неправильные цифры, пропущенные пункты, нарушенный формат. Включается автоматически в любую полную верификацию.
_Аналогия:_ проверка орфографии в Word — быстро находит очевидные ошибки, но не оценивает качество текста.
_Пример:_ М2 «Расширенная» проверяет точный перенос данных, выполнение запроса, суммы, границы значений и формат документа.

---

**Полная верификация (П1–П14)** — глубокая проверка, включающая смысловой анализ. Ищет логические ошибки, противоречия, слабые аргументы, скрытые допущения.
_Аналогия:_ рецензия эксперта на научную статью — не только грамматика, но и методология, логика, обоснованность выводов.
_Пример:_ П1 «Аналитик» проверяет обратную логику, скрытые допущения, полноту, стресс-тест выводов — всё, что важно для аналитических записок.

---

**Workflow (рабочий процесс)** — фиксированная последовательность шагов, по которой система проводит верификацию. В системе 5 этапов: от определения задачи до итогового отчёта.
_Аналогия:_ маршрутный лист в поликлинике — сначала регистратура, потом осмотр, потом анализы, потом заключение. Нельзя перескочить через этап.
_Пример:_ Этап 0 (определить задачу) → Этап 1 (подобрать пресет) → Этап 2 (контрольная точка) → Этап 3 (выполнить задачу) → Этап 4 (запустить верификацию) → Этап 5 (отчёт).

---

**Контрольная точка** — обязательная пауза перед запуском верификации, на которой система показывает план проверок и ждёт вашего одобрения. Пропуск контрольной точки запрещён.
_Аналогия:_ подтверждение заказа в интернет-магазине. Перед оплатой вы видите: что заказали, сколько стоит, куда доставить. Можете изменить или отменить. Без нажатия «Подтвердить» ничего не произойдёт.
_Пример:_ система показывает: «Рекомендую пресет П1 + скрипты format, references. Подтвердите для старта.» Вы отвечаете: «Добавь ещё проверку дат» или «Всё ок — начинай».

---

**Субагент** — отдельный AI-помощник, которому основной агент делегирует часть работы. Субагент-верификатор не видел, как документ создавался — он проверяет его «с чистого листа», как независимый аудитор.
_Аналогия:_ внешний рецензент. Автор статьи отдаёт её рецензенту, который не участвовал в написании — поэтому он видит ошибки, которые автор «не замечает».
_Пример:_ основной агент написал аналитическую записку → субагент-верификатор получает записку + исходные данные и проверяет её по списку механизмов.

---

**Оркестратор** — главный управляющий агент, который координирует работу нескольких субагентов и скриптов. Распределяет задачи, собирает результаты, разрешает расхождения.
_Аналогия:_ дирижёр оркестра. Каждый музыкант играет свою партию, а дирижёр следит, чтобы всё звучало вместе.
_Пример:_ при пресете П5 «Максимум» оркестратор запускает параллельно скрипты + трёх субагентов (Логик, Фактчекер, Редактор), а затем сводит их отчёты в единый результат.

---

**Reference-файл (справочный файл)** — документ с подробным описанием механизмов верификации: пошаговые процедуры, примеры, критерии оценки. Это «база знаний» системы.
_Аналогия:_ методичка для лаборанта. В ней написано: «Как проверить прочность: шаг 1 — закрепите образец, шаг 2 — приложите нагрузку...»
_Пример:_ файл `mechanisms_logical.md` содержит процедуры для 9 логических механизмов (скрытые допущения, парадоксы, обратная логика и др.).

---

**Severity (серьёзность находки)** — уровень критичности обнаруженной проблемы. Определяет, насколько срочно нужно реагировать.
_Аналогия:_ цвета светофора. Красный — стоп, нужно исправить. Жёлтый — внимание, желательно исправить. Зелёный — всё в порядке.
_Пример:_ 🔴 Критично — сумма в итоговой строке не совпадает с суммой слагаемых. 🟡 Существенно — вывод стоит на одной посылке (хрупкий). 🟢 Рекомендация — можно добавить ссылку на источник.

---

**Дрейф смысла** — постепенное, незаметное изменение смысла текста при многократных правках, пересказах или переводах. Каждое отдельное изменение кажется допустимым, но в сумме смысл искажается.
_Аналогия:_ «испорченный телефон». Каждый следующий участник чуть-чуть меняет фразу — и в конце она означает совсем другое.
_Пример:_ оригинал: «необходимо обеспечить» → правка 1: «следует обеспечить» → правка 2: «желательно обеспечить» → правка 3: «можно рассмотреть». Обязательное требование стало необязательной рекомендацией.

---

**Ложноположительное срабатывание (false positive)** — система сообщает об ошибке, но на самом деле всё правильно. Это «ложная тревога».
_Аналогия:_ пожарная сигнализация сработала от пара из чайника — пожара нет, но сигнал есть.
_Пример:_ скрипт verify_dates.py нашёл дату «2027 год» и пометил как ошибку, хотя в документе речь идёт о плановом прогнозе.

---

**Ложноотрицательное срабатывание (false negative)** — система пропустила реальную ошибку. Это «пропуск».
_Аналогия:_ охранник на проходной не заметил поддельный пропуск.
_Пример:_ механизм проверки противоречий не заметил, что в одном абзаце написано «рост на 5%», а в другом «снижение показателя» — потому что они разделены 10 страницами текста.

---

**Confirmation bias (предвзятость подтверждения)** — склонность находить подтверждения уже принятому решению и не замечать противоречия. Главная причина, почему AI не должен проверять собственную работу в том же контексте.
_Аналогия:_ перечитывание своего текста — ошибки «замыливаются», потому что вы знаете, что хотели написать.
_Пример:_ режим `self` (AI проверяет сам себя) менее надёжен, чем `subagent` (независимый проверяющий), именно из-за confirmation bias.
